# MYTコーチング：プロジェクト前提コンテキスト（Claude Code投入用）

> 参照元：`/mnt/data/MYT_zip/` 配下のマニュアル・テンプレ、および本プロジェクト内の確定事項（会話ログに基づく）。

## 1. 事業・サービスの全体像

### 1-1. 提供サービス（2本立て）
1) **技術×経営の統合診断（レポート付き）**  
- 目的：現状のボトルネック特定、優先順位設計、次の30日ToDo提示  
- 納品物（運用上の想定）：レポート（Google Docs → PDF化可）、ToDoリスト、優先順位表  
- 推奨：納品後のオンライン説明（60分）  
- 価格：**55,000円（税込）**  
- 初月の提供枠：**5件限定**（枠超過時は抽選運用）

2) **技術×経営コーチング（伴走）**  
- 目的：設計で終わらせず、実装・運用の定着まで伴走  
- 期間：**12ヶ月**（途中で**3ヶ月レビュー**を入れる）  
- 実施：週次または隔週のセッション（原則60分）＋月次KPI棚卸し  
- 実地（訪問）：**月1回まで**  
- 訪問エリア：**関西圏／関東圏**  
- 同時並行：**最大3社**  
- 価格帯（現時点のドラフト）：  
  - シーシャ部門：**5万円/月 × スタッフ数（オーナー含む）**  
  - 経営：**8万円/月**

### 1-2. 想定ターゲット
- シーシャ店：**新規開業準備中**／**既存店のリブランディング**  
- 「味（技術）」と「数字（経営）」を分断せず、統合的に改善したいオーナー

## 2. 顧客導線・運用フロー（標準）

### 2-1. 基本フロー（共通骨格）
**受注 → 審査 →（必要時）抽選 → 決済 → 提出 → 制作 → 納品**

### 2-2. ステータス管理（案件管理の共通ステータス案）
1) 受注（受付）  
2) 審査中  
3) 抽選待ち（必要時）  
4) 当選（決済待ち）  
5) 決済済  
6) 提出待ち  
7) 提出完了  
8) 制作中  
9) 納品準備  
10) 納品済  
11) 完了（フォロー済）  
12) キャンセル／失格（期限超過など）

### 2-3. 受付・審査・抽選の要点
- 受付チャネル：Instagram DM／X DM（推奨：キーワードで振り分け）
  - キーワード例：診断＝「統合診断」、コーチング＝「先行」「コーチング」
- 受付後は **DMで要件確定しない**（漏れ防止）。原則「申込フォーム」へ誘導。
- 審査目的：品質担保／ミスマッチ防止／リスク回避  
  - 通過優先の例：課題が具体、実行意欲あり、提出物を揃えられる  
  - 保留・不採択の例：丸投げ、期限遵守が怪しい、過度な値引き交渉、攻撃的態度
- 抽選（枠超過時のみ）：  
  - 抽選対象：審査通過者のみ  
  - 枠確定条件：**当選連絡から48時間以内の決済**（未決済は失効→繰上げ）

### 2-4. 決済・提出・納品の要点
- 決済：オンライン決済（Stripe決済リンク／請求書）を原則。例外で銀行振込（手数料は利用者負担）
- 制作開始条件：**決済済 + 提出完了**
- 提出期限の目安：**決済後7日以内**（診断／コーチングとも）
- 納品形態：PDF推奨（必要に応じて Notion / Google Docs）

## 3. オペレーション設計（Google Drive / Sheets中心）

> 重要：**MASTERは触らない。運用は必ずコピー（CLIENT）で行う。**

### 3-1. Driveフォルダ構成（推奨）
- `00_MASTER（触るな）`：MASTER本体隔離  
- `01_運用（普段ここ）`：ショートカット置き場、日常導線  
- `10_CLIENT/店名`：案件ごとの作業フォルダ  
- `90_納品`：PDFなど最終成果物置き場（任意）

### 3-2. ファイル命名（推奨）
- MASTER本体：`【MASTER】シーシャ診断テンプレ_v2`  
- ショートカット：`【使う】シーシャ診断テンプレ_v2（コピー作成用）`  
- CLIENTコピー：`【CLIENT】店名_診断_YYYY-MM-DD`

## 4. 診断テンプレ（分析用スプレッドシート）の構造

### 4-1. テンプレの役割
- **Tech_QA / Biz_QA の回答（○/×）** からギャップを抽出し、  
  **Scoreboard / Priority_Map / ToDo_4weeks / ThisWeek** に落とす。  
- Apps Script により「ギャップ→ToDo生成」を自動化。

### 4-2. シート構成（MASTERテンプレ：v2）
- `Config`：設定（週、日付など）
- `Report_Template`：レポート原稿の母体（出力の元）
- `Tech_QA`：技術質問（○/×入力）
- `Biz_QA`：経営質問（○/×入力）
- `Gaps_Auto`：ギャップ自動抽出
- `Scoreboard`：スコア可視化
- `Priority_Map`：優先度マップ（技術×経営を統合表示する想定）
- `ToDo_4weeks`：4週間ToDo（自動生成の主アウトプット）
- `ThisWeek`：今週分（Configの週指定で抽出）
- `Task_Library`：タスク辞書（ToDo生成の根拠データ）

### 4-3. ToDo生成の前提（Task_Libraryの重要列）
- `Trigger`：`Tech_QA/Biz_QA` のID（例：`TECH-01`）と **完全一致** するキー  
- `推奨Week` / `優先度（P0/P1/P2）` / `タスク名`：必須  
- 任意：`所要時間(分)` / `担当`（ToDoに反映したい場合）

### 4-4. 日常運用（診断テンプレ側）
- ショートカットを開き **コピー作成** → CLIENT名で保存 → `10_CLIENT` に移動
- `Tech_QA / Biz_QA` に **プルダウン（○/×）** で入力  
- メニュー：`診断ツール → ToDo生成（ギャップ→4週間）` 実行  
- `ToDo_4weeks` の4行目以降にタスクが入ることを確認  
- `Config` の「現在の週(1-4)」で `ThisWeek` を抽出  
- `ToDo_4weeks!B2` に `Week1開始日` を入れると期限列が自動計算

## 5. 顧客台帳（抽選運用対応）の構造

### 5-1. 設計思想（運用コスト最小化）
- **入力の正は `Clients` シートのみ（案件1行=顧客1件）**
- 当選者／請求対象／未入金／期限超過は **FILTER式で自動抽出**（読取専用）
- 候補リスト（担当者・ステータス等）は `Lists`（増やしてOK）
- Apps Script（台帳ツール）で「空行へ移動」「一覧更新」などをメニュー化

### 5-2. シート構成（台帳テンプレ：v3）
- `Dashboard`：全体サマリー（基本：自動）
- `Clients`：入力台帳（ここがマスタ）
- `Winners_当選者`：抽出（読取）
- `Invoice_請求対象`：抽出（読取）
- `Unpaid_未入金`：抽出（読取）
- `Overdue_期限超過`：抽出（読取）
- `Views`：ビュー
- `Lists`：候補リスト
- `Config`：設定
- `HowTo`：使い方

### 5-3. 日々の運用（台帳側の最短手順）
- (1) 申込受付：流入/連絡先/地域/申込日、ステータス=申込
- (2) 審査・抽選準備：審査ランク(A/B/C/D)、審査メモ、抽選期（例：2026-02）
- (3) 抽選結果確定：当選/次点/落選/対象外 → 当選者は `Winners_当選者` に自動表示
- (4) 当選連絡：当選連絡日、次アクション期限（決済締切など）
- 以降：請求→入金→提出→制作→納品まで、`Clients` の1行を前に進めるだけで一覧が追従

## 8. 既知の運用リスク（設計上の注意）
- **MASTER誤編集**：テンプレ破壊の最大リスク。隔離＋ショートカット運用が前提。
- **ID不一致によるToDo生成失敗**：`Tech_QA/Biz_QA` のIDと `Task_Library.Trigger` は完全一致が必須。
- **権限許可で停止**：初回のみ Apps Script の権限許可が必要（Googleの「未確認アプリ」導線を踏む）。
- **提出遅延**：提出物遅延＝納期後ろ倒し／精度低下の前提（運用手順書に明記）。
