# mode-a-run.md
## Claude実行テンプレ（単店の赤字改善 / single_store_turnaround）

> **Note**: このテンプレートを実行すると、`CLAUDE.md` に定義された共通ガードレール（P-hacking禁止ポリシー、出力品質ポリシー、実行プロトコル等）が自動的に適用されます。

### 使い方
1. 下の入力欄を埋める
2. まず「Step 1」だけ投げる（分析計画固定）
3. 次に「Step 2」暫定レポート
4. 最後に「Step 3」提出版整形

---

## 実行モード
- Mode: `single_store_turnaround`

## 目的
単店の赤字を構造分解し、4〜12週で改善可能な実行計画を作る。
P-hackingを避けるため、分析計画を先に固定する。

## Step 1（最初にこれだけ投げる）
以下の案件データを使って、**最終レポートはまだ作らず**、まず次の4点だけ出力してください。
P-hacking防止のため、分析計画を先に固定してください。

1. 分析計画（P-hacking防止）
   - 主要KPI（結果指標 / 先行指標）
   - 分析期間・比較基準
   - セグメント（時間帯 / 曜日 / 商品 / 必要ならスタッフ）
   - 外れ値処理ルール
   - 検証的分析 vs 探索的分析の境界
2. データ不足・矛盾・確認事項（重要度順）
3. 現時点の分析可能範囲
4. 追加取得データTOP10（取得優先順位つき）

※禁止：結果を見てから都合よくKPIや切り口を変更すること

---

## Step 2（暫定版レポート作成）
不足データがあっても分析を止めず、**仮説を明示**したうえで暫定版の経営レポートを作成してください。
以下を必ず含めてください。

- 検証的分析（事前固定した分析）
- 探索的分析（追加で見つけた示唆）
- 赤字の主因（最大3つ）
- 72時間以内の止血施策
- 1〜4週の短期施策
- 1〜3ヶ月の中期施策
- 30/60/90日アクションプラン
- 今やらないこと（理由付き）
- レポートの限界

特に重視する観点：
- 売上分解（客数 / 客単価 / 時間帯 / 曜日）
- 粗利改善余地（フレーバー / 炭 / ドリンク / 消耗品 / ロス）
- 人件費効率（シフト × 売上）
- 品質ばらつきの構造要因
- オーナー依存のリスク

---

## Step 3（クライアント提出用に整形）
上記の暫定版を、クライアント提出用に再編集してください。要件は以下です。

- 読みやすさ重視（見出し・箇条書き・必要最小限の表）
- 過度に攻撃的な表現は避ける
- ただし論点の厳しさは維持する
- エグゼクティブサマリーを先頭に置く
- 優先順位（1〜5）を明記
- 「今やらないこと」を明記
- P-hacking防止の分析計画サマリーを付録または冒頭に記載
- 最後に「本レポートの限界」を記載
