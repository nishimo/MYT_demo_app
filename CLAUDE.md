## Claude Code Skill: Shisha Store Turnaround & Integrated Tech x Management Diagnostic (No P-hacking)

### 概要
このスキルは、シーシャ店向けの経営レポートをクライアント提出品質で作成するための Claude Code 用スキルです。
以下の2モードを提供します。

- `single_store_turnaround`：単店の赤字改善（4〜12週の立て直し）→ `mode-a-run.md` を使用
- `integrated_tech_management_diagnostic`：シーシャ技術評価 × 経営評価の統合診断 → `mode-b-run.md` を使用

両モード共通で、**P-hacking（都合の良い分析切り取り）を禁止**します。

---

## 目的
- 数字だけ/感覚だけの診断を避ける
- 品質・オペレーション・収益性を接続して分析する
- クライアント提出に耐える構造化レポートを作る
- 施策に優先順位・期限・KPIを付与する
- P-hackingを防ぎ、再現性ある判断を行う

---

## モード定義

### Mode A: single_store_turnaround
**用途**：単店の赤字改善、資金繰り改善、短中期の立て直し
**実行テンプレート**：`mode-a-run.md`

#### 主な論点
- 赤字の構造分解（売上、粗利、原価、人件費、ロス）
- 72時間以内の止血施策
- 30/60/90日改善計画
- 品質ばらつきがリピート・客単価に与える影響
- オーナー依存の解消

---

### Mode B: integrated_tech_management_diagnostic
**用途**：技術評価（シーシャ品質）と経営評価を統合して診断
**実行テンプレート**：`mode-b-run.md`

#### 主な論点
- 技術品質の再現性とばらつき
- 品質→体験→再来店→利益の接続仮説
- スタッフ/時間帯/店舗の差異
- 教育・標準化・配置・評価制度への落とし込み
- 技術改善の経営インパクト優先順位

## 共通ガードレール（必須）

### 1. P-hacking禁止ポリシー
分析前に以下を固定すること（Pre-specification）。

- 主要KPI（結果指標 / 先行指標）
- 分析期間
- 比較基準（前年同月 / 前月 / 予算 / 店舗比較 等）
- セグメント（時間帯 / 曜日 / 商品 / スタッフ 等）
- 成功判定基準（何をもって改善とするか）
- 外れ値処理ルール（除外条件）

#### 禁止事項
- 結果を見てからKPI・期間・切り口を都合よく変更する
- 都合の悪い日/データだけを後から除外する
- 探索的示唆を確定事実のように書く
- 相関だけで因果を断定する

### 2. 分析区分の明示（必須）
出力では必ず分けること。
- **検証的分析（Confirmatory）**：事前固定の分析
- **探索的分析（Exploratory）**：追加で見つけた示唆

### 3. 不確実性の明示（必須）
- サンプル数不足
- データ欠損
- 紐づけ不能（例：技術評価と売上の接続不可）
- 仮説依存論点
は必ず記載する。

## 共通実行プロトコル（Claude Code向け）

### Phase 0: 分析計画固定（P-hacking防止）
最初に以下のみ出力（最終レポートはまだ作らない）
1. 分析目的
2. 主要KPI（結果指標 / 先行指標）
3. 分析期間・比較基準
4. セグメント
5. 外れ値処理ルール
6. データ制約
7. 検証的分析 / 探索的分析の境界

### Phase 1: データ不足・矛盾チェック
- 不足データ（重要度順）
- 矛盾/不整合
- 分析精度への影響
- 追加取得推奨データ
- 現時点の分析可能範囲

### Phase 2: 事実ベース分析
- 業績推移
- 売上分解
- 粗利/原価分解
- 人件費/運営効率
- 品質/提供/接客の構造論点

### Phase 3: 真因仮説（反証可能性付き）
- 表面課題
- 真因仮説
- 根拠
- 反証可能性
- 検証方法

### Phase 4: 改善提案
- 短期（1〜4週）
- 中期（1〜3ヶ月）
- 長期（3〜12ヶ月）
各施策に以下を付与：
- 目的 / 実施内容 / 実施手順 / 工数 / 期待効果 / リスク / KPI / 優先度 / 期限

### Phase 5: 提出版整形
- エグゼクティブサマリー
- 優先順位
- 30/60/90日計画
- 意思決定ポイント（3つ）
- 今やらないこと
- レポートの限界
- 分析計画サマリー（P-hacking防止）

### 期待成果物

## Mode B の診断観点（技術×経営統合）

### 技術評価の標準観点（例）
- セッティング準備（衛生、段取り、速度）
- パッキング品質（詰め方、量、通り、特性適合）
- 熱管理（立ち上がり、安定性、焦げ抑制）
- 味/煙品質（明瞭さ、煙量、滑らかさ、再現性）
- 提供オペレーション（提供時間、説明、フォロー）
- 顧客体験翻訳（初心者配慮、提案力、不満予防）

### 経営評価の標準観点（例）
- 売上（客数/単価/頻度）
- 粗利（原価・ロス）
- 人件費率/シフト効率
- 席稼働・滞在・回転
- クレーム率/再来店/紹介
- 教育コスト/定着
- オーナー依存度

### 統合診断フレーム（必須）
技術/運営の観察事実
→ 顧客体験への影響仮説
→ 行動指標（追加注文・再来店・レビュー等）への影響仮説
→ 売上/粗利/人件費効率への影響仮説
→ 検証方法（追加で測るべきデータ）
